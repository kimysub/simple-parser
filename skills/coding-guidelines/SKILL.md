# Coding Guidelines Skill

Andrej Karpathy의 LLM 코딩 관찰에서 파생된 행동 가이드라인.

## 적용 시점

이 스킬은 다음 상황에서 자동으로 적용된다:
- 새로운 코드 작성 시
- 기존 코드 수정 시
- 코드 리뷰 시
- 리팩토링 시
- 버그 수정 시

## 4가지 핵심 원칙

### 원칙 1: 코딩 전에 생각하라

구현 전에 반드시:
1. 가정을 명시적으로 진술한다.
2. 여러 해석이 가능하면 모두 제시한다.
3. 더 간단한 접근이 있다면 제안한다.
4. 불명확한 점이 있으면 멈추고 질문한다.

**잘못된 예:**
```python
# 사용자가 "데이터 내보내기 추가"를 요청했을 때
# 조용히 JSON+CSV 모두 구현, 파일 위치 임의 결정
def export_users(format='json'):
    users = User.query.all()  # 모든 사용자? 페이지네이션은?
    with open('users.json', 'w') as f:  # 파일 위치를 왜 여기로?
        json.dump([u.to_dict() for u in users], f)
```

**올바른 예:**
```
구현 전에 확인이 필요합니다:
1. 범위: 전체 사용자? 필터된 부분집합? (프라이버시 고려)
2. 형식: 파일 다운로드? API 응답? 백그라운드 작업?
3. 필드: 어떤 사용자 필드를 포함? (민감한 데이터 확인)
가장 단순한 접근: 페이지네이션된 JSON을 반환하는 API 엔드포인트.
어떤 방식을 원하시나요?
```

### 원칙 2: 단순함 우선

- 요청받은 것만 구현한다.
- 단일 용도에 추상화를 쓰지 않는다.
- "나중에 필요할 수도" 있는 기능은 나중에 추가한다.

**잘못된 예:** 할인 계산 요청에 Strategy 패턴 + Factory + Config 시스템 구현 (200줄)

**올바른 예:**
```python
def calculate_discount(amount: float, percent: float) -> float:
    return amount * (percent / 100)
```

### 원칙 3: 수술적 변경

변경된 모든 줄이 사용자의 요청에 직접 연결되어야 한다.

하지 말 것:
- 버그 수정 중 따옴표 스타일 변경
- 로깅 추가 중 타입 힌트 추가
- 인접 코드의 포매팅 "개선"

할 것:
- 기존 스타일 유지 (따옴표, 들여쓰기, 네이밍 규칙)
- 요청된 변경만 수행
- 내 변경으로 발생한 고아 코드만 정리

### 원칙 4: 목표 기반 실행

명령형 작업을 선언적 목표로 변환한다:

| 대신에... | 이렇게 변환... |
|-----------|---------------|
| "검증 추가" | "잘못된 입력 테스트 작성 → 통과시키기" |
| "버그 수정" | "재현 테스트 작성 → 통과시키기" |
| "리팩토링" | "기존 테스트 통과 확인 → 리팩토링 → 재확인" |

다단계 작업은 검증 가능한 계획을 세운다:
```
1. [단계] → 검증: [체크]
2. [단계] → 검증: [체크]
3. [단계] → 검증: [체크]
```

## 핵심 인사이트

> "좋은 코드는 오늘의 문제를 단순하게 해결하는 코드이지, 내일의 문제를 성급하게 해결하는 코드가 아니다."

> "LLM은 특정 목표에 도달할 때까지 반복하는 것에 뛰어나다. 무엇을 하라고 지시하지 말고, 성공 기준을 주고 달리게 하라." — Andrej Karpathy
