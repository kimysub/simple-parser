# CLAUDE.md

프로젝트 전반에 적용되는 Claude Code 행동 가이드라인.
Andrej Karpathy의 LLM 코딩 관찰에서 파생된 원칙을 기반으로 한다.

> **Trade-off:** 이 가이드라인은 속도보다 신중함에 무게를 둔다. 사소한 작업에는 판단력을 사용할 것.

---

## 핵심 원칙

### 1. 코딩 전에 생각하라 (Think Before Coding)

**가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현 전에:
- 가정을 명시적으로 진술하라. 불확실하면 물어라.
- 여러 해석이 가능하면 제시하라 — 조용히 하나를 선택하지 마라.
- 더 간단한 접근이 있다면 말하라. 필요시 반박하라.
- 뭔가 불분명하면 멈춰라. 뭐가 헷갈리는지 말하고 물어라.

### 2. 단순함 우선 (Simplicity First)

**문제를 해결하는 최소한의 코드. 추측성 코드 금지.**

- 요청받지 않은 기능 추가 금지.
- 단일 용도 코드에 추상화 금지.
- 요청받지 않은 "유연성"이나 "설정 가능성" 금지.
- 불가능한 시나리오에 대한 에러 핸들링 금지.
- 200줄이 50줄로 가능하면 다시 작성하라.

자문하라: "시니어 엔지니어가 이게 과하다고 할까?" 그렇다면 단순화하라.

### 3. 수술적 변경 (Surgical Changes)

**필요한 것만 수정하라. 자기가 만든 것만 정리하라.**

기존 코드 편집 시:
- 인접 코드, 주석, 포매팅을 "개선"하지 마라.
- 고장나지 않은 것을 리팩토링하지 마라.
- 기존 스타일을 따라라, 다르게 하고 싶어도.
- 관련 없는 데드 코드를 발견하면 언급만 하고 삭제하지 마라.

내 변경이 고아(orphan)를 만들었을 때:
- 내 변경으로 사용되지 않게 된 import/변수/함수는 제거하라.
- 기존에 있던 데드 코드는 요청 없이 제거하지 마라.

**테스트:** 변경된 모든 줄은 사용자의 요청에 직접 연결되어야 한다.

### 4. 목표 기반 실행 (Goal-Driven Execution)

**성공 기준을 정의하라. 검증될 때까지 반복하라.**

작업을 검증 가능한 목표로 변환하라:
- "검증 추가" → "잘못된 입력에 대한 테스트 작성 → 통과시키기"
- "버그 수정" → "재현 테스트 작성 → 통과시키기"
- "X 리팩토링" → "전후로 테스트가 통과하는지 확인"

다단계 작업에서는 간략한 계획을 세워라:
```
1. [단계] → 검증: [체크]
2. [단계] → 검증: [체크]
3. [단계] → 검증: [체크]
```

## 코드 품질 기준

### 언어별 관례
- 해당 언어와 프레임워크의 관용적 패턴을 따른다.
- 프로젝트에 이미 존재하는 패턴이 있다면 그것을 우선한다.
- 새 패턴을 도입하기 전에 기존 코드베이스를 먼저 확인한다.

### 에러 핸들링
- 에러는 가능한 발생 지점에 가까이서 처리한다.
- 에러 메시지는 구체적이고 실행 가능하게 작성한다.
- 조용히 실패하지 않는다 — 로깅하거나 전파한다.

### 테스트
- 변경사항에 대해 테스트가 존재하는지 확인한다.
- 버그 수정 시 재현 테스트를 먼저 작성한다.
- 테스트는 하나의 동작을 검증한다 — 여러 관심사를 섞지 않는다.

### 커밋과 변경
- 각 커밋은 하나의 논리적 변경을 담는다.
- 커밋 메시지는 "왜"를 설명한다, "무엇"이 아니라.
- 관련 없는 변경을 하나의 커밋에 넣지 않는다.

## 작업 패턴

### 새 기능 구현 시
1. 요구사항의 가정을 명시하고 확인한다.
2. 가장 단순한 구현 방안을 먼저 제안한다.
3. 테스트를 먼저 작성하거나, 수동 검증 방법을 정한다.
4. 구현 후 검증한다.

### 버그 수정 시
1. 버그를 재현하는 테스트를 먼저 작성한다.
2. 최소한의 수정으로 해결한다.
3. 테스트가 통과하는지 확인한다.
4. 회귀가 없는지 기존 테스트를 돌린다.

### 리팩토링 시
1. 기존 테스트가 통과하는지 확인한다.
2. 리팩토링을 수행한다.
3. 동일한 테스트가 여전히 통과하는지 확인한다.
4. 동작 변경 없이 구조만 개선됐는지 확인한다.

## 문서 자동 업데이트 (Auto-update Documentation)

코드 변경 후 아래 문서를 자동으로 업데이트한다. 사용자에게 물어볼 필요 없다.

| 파일 | 내용 |
|---|---|
| `README.md` | 기능, 설치, 사용법, 프로젝트 구조 |
| `QUICK_START.md` | 빠른 시작 가이드 (단계별) |
| `Plan.md` | 아키텍처, 의존성, 구현 단계 |
| `Checklist.md` | 단계별 상태, 테스트 수, 검증 명령어 |
| `Implemented_Feature.md` | 구현된 기능 상세 설명 |

규칙:
- 새 기능 추가, API 변경, 구조 변경 시 관련 문서를 함께 업데이트한다.
- 문서에 변경 사항이 정확히 반영되어야 한다 (테스트 수, 엔드포인트, 파일 구조 등).
- 코드 변경과 문서 업데이트는 같은 작업의 일부로 처리한다.

## 금지 사항 (Anti-patterns)

| 하지 마라 | 대신 이렇게 하라 |
|-----------|-----------------|
| 가정을 조용히 하고 구현 | 가정을 명시하고 확인 받기 |
| 단일 할인 계산에 Strategy 패턴 | 실제로 필요할 때까지 단일 함수 유지 |
| 버그 수정하면서 인용부호 스타일 변경 | 보고된 문제만 수정 |
| "코드를 검토하고 개선하겠습니다" | "버그 X에 대한 테스트 작성 → 통과 → 회귀 없음 확인" |
| 요청 없이 주석/타입힌트 추가 | 기존 스타일 유지 |
| 미래를 위한 추상화 | 오늘의 문제를 단순하게 해결 |

## 이 가이드라인이 작동하고 있다면

- diff에 불필요한 변경이 줄어든다.
- 과도한 구현으로 인한 재작성이 줄어든다.
- 구현 전에 명확화 질문이 온다 (실수 후가 아니라).
- PR이 깔끔하고 최소한이다 — 드라이브바이 리팩토링 없음.
