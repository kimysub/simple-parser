# 코드 리뷰

변경된 코드에 대해 시니어 엔지니어 관점에서 코드 리뷰를 수행한다.

## 리뷰 절차

1. `git diff` 또는 `git diff --staged`로 변경사항을 확인한다.
2. 다음 관점에서 리뷰한다:

### 체크리스트
- **정확성**: 로직이 올바른가? 엣지 케이스는 처리되었는가?
- **단순성**: 불필요한 복잡성이 있는가? 더 단순한 방법이 있는가?
- **수술적 변경**: 요청과 무관한 변경이 포함되어 있는가?
- **네이밍**: 변수, 함수, 클래스 이름이 의도를 명확히 전달하는가?
- **에러 핸들링**: 실패 케이스가 적절히 처리되었는가?
- **테스트**: 변경에 대한 테스트가 있는가? 있다면 충분한가?
- **보안**: 인젝션, 인증, 인가 문제가 있는가?
- **성능**: 명백한 성능 문제가 있는가? (N+1, 불필요한 루프 등)

3. 발견사항을 심각도별로 분류한다:
   - 🔴 **Must Fix**: 버그, 보안 문제, 데이터 손실 가능성
   - 🟡 **Should Fix**: 코드 품질, 유지보수성, 잠재적 문제
   - 🟢 **Nice to Have**: 스타일, 개선 제안

4. 각 발견사항에 대해 구체적인 수정 방안을 제안한다.

## 입력
$ARGUMENTS
